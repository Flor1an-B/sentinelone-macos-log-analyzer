## Opening the Report

The HTML report is a **single self-contained file** — no internet connection required, no external fonts or scripts. Open it in any modern browser:

```bash
# macOS / Linux
open SentinelLog_2026.03.19_17.40.14_root_report/SentinelLog_2026.03.19_17.40.14_root_report.html

# Windows
start SentinelLog_2026.03.19_17.40.14_root_report\SentinelLog_2026.03.19_17.40.14_root_report.html
```

## Analysis Period Filter

The **Analysis Period** buttons in the sidebar filter all time-sensitive sections of the report without reloading:

| Button | Scope | Use when |
|---|---|---|
| **24h** | Last 24 hours relative to the dump | Investigating a recent incident or verifying current state |
| **7 days** | Last 7 days | Weekly threat review |
| **30 days** | Last 30 days | Monthly posture assessment |
| **All** | Entire dump history | Full forensic review (default) |

When a period is selected, the following update automatically:

- All section content (Quick Brief, Findings, Timeline, IOC, etc.)
- **Risk score** gauge and bar
- **Security Risk** score in the sidebar
- **Findings badge** count on the navigation link
- Severity chip counts in the hero header

> **Note:** Static sections — System Context, Network, Performance, Services, Agent Health — always show current snapshot data regardless of the selected period.

## Section-by-Section Guide

### Overview

#### Hero
The top card shows:
- **Risk gauge** (circular arc, 0–100) and **risk bar** — animated, updates with period
- **Severity chips** (Total, Critical, High, Medium, Low, Info) — update with period
- **System identity** (hostname, model, OS, user, S1 agent version, dump date)
- **Agent Health** indicator with reasons if degraded

#### Operational Alerts
Auto-generated prioritized alerts based on parsed system state (independent of behavioral detections):

| Level | Color | Meaning |
|---|---|---|
| CRITICAL | Red | Immediate action required — protection gap or active bypass |
| HIGH | Orange | Significant issue — investigate promptly |
| MEDIUM | Yellow | Notable configuration issue — review when possible |
| INFO | Blue | Informational — no immediate action required |

Each alert includes a **Detail** (what was detected and why it matters) and an **Action** (specific remediation steps).

### System Group

#### System Context
Hardware and OS identity: model, serial number, architecture, OS version, primary user, CPU count, SIP state, boot arguments.

**What to look for:** SIP disabled (red badge), unusual boot arguments (`amfi_get_out_of_my_way`, `-no_compat_check`).

#### Performance
CPU load averages, memory pressure (free/wired/compressed MB), disk usage, battery/power state.

**What to look for:** Memory pressure `CRITICAL` or `WARNING` (can cause the agent to be killed by the OS), disk above 90%.

#### Network
Active network interfaces (IPv4/IPv6/MAC), open connections from `lsof -i` and `netstat`, DNS servers, proxy configuration.

**What to look for:** Unexpected outbound connections on unusual ports, proxy configured pointing to an internal MITM.

#### Services
Launch daemons, launch agents, third-party services (enabled/disabled), kernel extensions.

**What to look for:** Unknown daemons or agents without a known vendor, unsigned kernel extensions.

#### System Activity *(period-sensitive)*
Boot/shutdown events, package installations, login/logout events within the selected period.

**What to look for:** Unexpected software installations (especially with `uid=0`), unusual boot frequency, manual installs from non-App-Store sources.

### SentinelOne Agent Group

#### S1 Agent Health
- **Daemon states** — each SentinelOne daemon with ready/not-ready status
- **Asset signatures** — integrity status of detection assets (valid / invalid / empty)
- **Database health** — state.db size, state.wonky presence, DB read/write GiB
- **Last full disk scan** status
- **Exclusions** — path exclusions and Deep Visibility exclusions in effect

**What to look for:** `invalid` asset signatures (especially `signatures`, `sha256` — detection gap), `state.wonky` present (DB crash recovery), scan never performed, large exclusion lists.

#### Comm. Analysis *(period-sensitive)*
- Management console connectivity (connected / last seen)
- RCP (remote command) request history with types and timestamps
- Keep-alive event frequency and recency
- Deep Visibility configuration errors
- Agent log error breakdown by component

**What to look for:** Agent disconnected from management, high error rate in specific components, DV config errors (version mismatch), missing keep-alive events (agent silently down).

---

### Security Analysis Group

#### Processes *(period-sensitive)*
Process profiles: each unique process seen in behavioral detections, with:
- Total event count and unique event types
- Linked MITRE techniques
- First/last seen timestamps
- Sample target paths

**What to look for:** Processes with many distinct event types (broad activity), unexpected binaries (paths outside `/usr`, `/Applications`, `/System`), processes triggering both recon and persistence events.

#### Findings *(period-sensitive)*
All rule findings for the selected period, sorted by severity. Each finding shows:
- Rule ID and MITRE technique (linked to attack.mitre.org)
- Severity badge
- Description of what was detected
- Recommendation (remediation steps)
- Evidence events with timestamps (expandable)

Use the **severity filter buttons** and **search bar** to focus on specific findings.

#### IOC *(period-sensitive)*
Indicators of Compromise extracted from behavioral detections:
- Suspicious process paths
- Unusual target files or directories
- Network addresses observed in events
- Group IDs for correlation

#### Timeline *(period-sensitive)*
Chronological view of all events in the selected period. Useful for establishing attack sequence and correlating activity across processes.

**What to look for:** Clusters of events in a short window (rapid recon or automated tool), events at unusual times (outside business hours), events immediately before or after known incidents.

#### Statistics *(period-sensitive)*
- **Detection heatmap** — activity by day and hour
- **Top processes** — by event count
- **Event type distribution** — what kinds of behavioral events were recorded
- **Daily event trend** — volume over time

#### Threat Intel
Threat intelligence asset versions from the SentinelOne agent:
- Dynamic Engine version and update date
- Static Signatures version and update date
- Static AI Library version and update date

**What to look for:** Outdated intelligence versions (gap between last update and dump date suggests the agent couldn't reach the update server).

#### Blind Spots
Coverage gaps automatically identified based on the parsed configuration:
- Daemons not ready (missing coverage)
- Assets with invalid signatures
- Missing Full Disk Access
- Disabled Deep Visibility features
- Large exclusion lists

## Triage Workflow

### L1 — Initial triage (5–10 minutes)

1. Check **Operational Alerts** — any CRITICAL or HIGH alerts require immediate escalation
2. Read **Quick Brief** — the 1-page summary is designed for rapid L1 assessment
3. Note the **Risk Score** and **Agent Health** indicator
4. If CRITICAL/HIGH findings are present → escalate to L2

### L2 — Detailed investigation (20–60 minutes)

1. Review **S1 Agent Health** — confirm the agent is fully operational and assets are valid
2. Review **Comm. Analysis** — confirm the agent is connected to management
3. Review **Findings** (filtered to HIGH+) — read descriptions and linked evidence
4. Review **Processes** — identify suspicious binaries
5. Use the **Timeline** to reconstruct the sequence of events
6. Cross-reference **IOC** with threat intelligence feeds
7. Check **Blind Spots** for coverage gaps that may have allowed undetected activity

### Changing analysis period

- Start with **All** for full picture
- Switch to **24h** or **7 days** to focus on recent activity
- The risk score, findings count, and all section content update automatically

## Risk Score Interpretation

| Score | Label | Recommended action |
|---|---|---|
| 0 | MINIMAL | No findings. Verify agent health and scan status. |
| 1–24 | LOW | Minor issues. Review LOW findings, monitor. |
| 25–49 | MEDIUM | Notable activity. Investigate MEDIUM+ findings. |
| 50–74 | HIGH | Significant threat indicators. Prioritize investigation. |
| ≥ 75 | CRITICAL | Active compromise indicators. Immediate response required. |

> The risk score is additive and capped at 100. A single CRITICAL finding (25 pts) combined with three HIGH findings (30 pts) yields a score of 55 (HIGH). The score does not represent certainty of compromise — it represents the weight of detected signals.
